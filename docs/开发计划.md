# 开发计划

这个文档合并原来的 `plans` 和 `specs`，只保留正常开发需要的信息：项目目标、架构思路、已完成内容、下一步计划和验收方式。它不写成逐行代码施工清单，具体实现继续通过小 PR 推进。

## 项目目标

AI PR Review 助手是一个网页端代码评审辅助工具。用户输入 GitHub PR 链接后，系统获取 PR 信息和代码变更，生成便于阅读的变更总结、风险提示和 Review 建议。

它不替代人工 Review，而是提前完成信息整理、风险初筛和建议归纳，让开发者把注意力放在真正需要判断的代码上。

## 当前能力

当前已经跑通最小链路：

1. 前端输入 GitHub PR 链接。
2. 后端校验并解析 PR URL。
3. 后端调用 GitHub REST API 获取公开 PR 元信息。
4. 前端展示 PR 标题、作者、状态、分支、增删行和变更文件数量。

当前还没有接入 diff 获取、风险识别和 AI Review 生成。

## 架构思路

整体流程：

```text
浏览器
  -> POST /api/analyze-pr
    -> 解析 GitHub PR URL
    -> 获取 PR 元信息
    -> 获取变更文件和 patch
    -> 构建分析上下文
    -> 执行规则风险预检查
    -> 调用 AI Review 服务
    -> 聚合结构化报告
  <- Review 报告 JSON
```

后端使用 FastAPI，模块按职责拆分：

- `pr_url_parser`：校验并解析 GitHub PR 链接。
- `github_client`：访问 GitHub REST API，并把响应映射成内部数据结构。
- `schemas`：定义请求和响应结构。
- `main`：组织 HTTP 路由、错误转换和模块调用。

后续会继续增加：

- `diff_collector`：获取 changed files 和 patch。
- `risk_engine`：执行确定性风险规则。
- `review_service`：组织上下文、调用模型并校验输出。
- `markdown_exporter`：生成可复制的 Review 文本。

前端使用 React + Vite + TypeScript，优先做成实用工具界面。当前页面只展示 PR 元信息，后续会逐步增加变更文件、风险项、Review 建议和 Markdown 导出。

## AI Review 思路

AI 层面向 OpenAI-compatible API 设计，不把业务逻辑绑定到某个固定模型服务。

计划使用环境变量配置：

- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`

模型输入采用分层上下文，不直接塞完整仓库：

- PR 标题和描述。
- 变更文件列表。
- 每个文件的 patch。
- 文件状态、新增行数、删除行数。
- 规则风险预检查结果。
- 必要时补充变更行附近的函数或类上下文。

模型输出应使用结构化 JSON。后端负责校验、归一化和错误处理，前端不直接消费自由文本。

## Review 质量控制

AI Review 的输出必须区分“确定问题”和“需要人工确认的问题”。

设计约束：

- 规则引擎发现的风险信号和模型生成的 Review 建议分开展示。
- 每条建议包含严重程度和置信度。
- 低置信度内容进入 Review 问题区，不直接当成缺陷。
- 推荐测试和代码问题分开展示。
- 聚合阶段去重相似建议，减少噪声。

## 下一阶段

### 3. 获取变更文件和 patch

目标：

- 调用 GitHub API 获取 PR changed files。
- 保存每个文件的路径、状态、新增删除行数和 patch。
- 对文件数量和 patch 大小设置上限，避免一次请求过大。

验收标准：

- 后端可以返回 PR 文件变更列表。
- 测试覆盖正常响应、空 patch、GitHub 错误和超限场景。
- 前端能展示变更文件列表的基础信息。

### 4. 规则化风险预检查

目标：

- 在 AI 调用前先用确定性规则识别明显风险。
- 风险规则先覆盖高价值场景，例如配置变更、鉴权相关文件、数据库迁移、测试删除、大规模变更和依赖变更。

验收标准：

- 每条风险包含文件路径、风险类型、原因和建议关注点。
- 规则测试使用本地 fixture，不依赖真实 GitHub 网络。
- 页面能把风险文件和普通变更文件区分展示。

### 5. Review 报告结构

目标：

- 定义统一的 Review 报告 schema。
- 报告包含摘要、整体风险等级、关键变更、风险文件、发现项、Review 问题和推荐测试。
- 前后端围绕同一套结构开发，减少后续 AI 接入时的返工。

验收标准：

- 后端 schema 有测试保护。
- 前端能用 fixture 渲染完整报告。
- 低置信度内容和确定问题在结构上分开。

### 6. AI Review 服务

目标：

- 接入 OpenAI-compatible API。
- 使用环境变量配置模型服务。
- 把 PR 元信息、patch 和风险预检查结果组装成紧凑上下文。
- 要求模型返回结构化 JSON，并在后端校验。

验收标准：

- 缺少 API key 时返回清楚错误。
- 模型超时或 JSON 无效时有可理解的失败提示。
- 测试中使用 fake provider，不依赖真实模型。
- 至少准备一个真实 PR 的手动验证记录。

### 7. 报告展示和 Markdown 导出

目标：

- 前端展示完整 Review 报告。
- 支持复制 Markdown，方便粘贴到 GitHub PR 评论区。

验收标准：

- 报告区域按摘要、风险、建议和测试分块。
- Markdown 内容和页面展示保持一致。
- 前端构建通过，并完成一次手动浏览器验证。

## 日常开发流程

1. 从 `main` 创建功能分支。
2. 明确本次 PR 只做一个主题。
3. 先写或调整测试。
4. 实现最小代码。
5. 跑后端测试和前端构建。
6. 更新相关文档。
7. 推送分支并创建 PR。
8. 手动测试通过后合并回 `main`。

## 暂不处理

这些能力先不进入近期计划：

- 多用户登录。
- 团队仪表盘。
- Review 历史数据库。
- 自动把评论发布回 GitHub。
- GitHub App 安装流程。

等基础 Review 链路稳定后，再判断是否需要继续扩展。
