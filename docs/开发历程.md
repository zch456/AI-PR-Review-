# 开发历程

这个项目是按“小功能、可运行、可验证”的方式推进出来的。每一阶段都围绕一个明确目标交付，并在完成后合并回 `main`。

## 1. 项目骨架与最小闭环

最开始先建立 React + Vite 前端和 FastAPI 后端，让项目能跑起来，而不是一开始就接 AI。

这一阶段完成：

- 后端 `/api/health` 健康检查。
- GitHub PR URL 解析。
- `/api/analyze-pr` 基础接口。
- 前端输入 PR 链接并展示解析结果。
- README 中记录后端、前端启动方式。

这个阶段的价值是把“网页输入 -> 后端处理 -> 前端展示”的链路跑通，为后续功能提供稳定骨架。

## 2. GitHub PR 元信息获取

第二阶段接入 GitHub REST API，获取公开 PR 的基础信息。

这一阶段完成：

- 新增 `github_client`。
- 获取 PR 标题、作者、状态、base/head 分支、增删行和变更文件数量。
- 后端接口从“只解析 URL”升级为“获取真实 PR 元信息”。
- 前端展示 PR 元信息。
- 使用 `httpx.MockTransport` 测试 GitHub 响应映射，避免测试依赖真实网络。

这个阶段开始让工具具备真实项目价值：用户输入一个公开 PR，能看到可靠的基础上下文。

## 3. 变更文件与 patch 获取

第三阶段继续从 GitHub API 获取 changed files。

这一阶段完成：

- 获取每个变更文件的路径、状态、新增行数、删除行数、总变更行数和 patch。
- `/api/analyze-pr` 返回 `files` 字段。
- 前端展示文件列表和 patch 可用状态。
- README 同步更新当前能力。

这个阶段是 AI Review 的关键前置能力，因为后续风险识别和模型分析都依赖文件级 diff。

## 4. 规则化风险预检查

第四阶段没有急着接模型，而是先加入确定性规则，减少完全依赖 AI 的不稳定性。

这一阶段完成：

- 新增 `risk_engine`。
- 识别依赖变更、配置变更、鉴权/安全相关文件、测试覆盖减少、单文件大变更和疑似敏感信息。
- `/api/analyze-pr` 返回 `overallRisk` 和 `riskSignals`。
- 前端展示整体风险和风险提示列表。
- 修复一次误报：依赖名里包含 `token` 不再直接判定为密钥。

这个阶段的价值是把“AI 之前的确定性判断”沉淀下来。规则输出可解释、可测试，也能给后续模型提供上下文。

## 5. AI Review 报告生成

第五阶段接入 DeepSeek，生成结构化 Review 报告。

这一阶段完成：

- 新增 `ai_reviewer`。
- 构建包含 PR 元信息、文件列表和 patch 的 prompt。
- 限制输入字符数，并按文件变更量优先选择上下文。
- 要求模型返回 JSON，后端负责解析、校验和归一化。
- 新增 `/api/generate-review`。
- 前端支持生成 AI Review、展示总结和发现项。
- 支持导出 Markdown 报告。

这一阶段让工具从“信息展示工具”升级为“Review 辅助工具”。

## 6. 配置和安全收口

AI 接入过程中出现过一次高风险问题：配置文件里曾出现硬编码 API Key 默认值。后续已修复为只从环境变量读取。

当前配置方式：

- `DEEPSEEK_API_KEY`：必须由用户环境变量提供。
- `DEEPSEEK_BASE_URL`：默认 `https://api.deepseek.com`。
- `DEEPSEEK_MODEL`：默认 `deepseekv4pro`，可通过环境变量覆盖。

这一步提醒后续开发：所有涉及凭据、token、API Key 的代码都要优先做安全检查。

## 当前成果

项目现在已经具备初步完整闭环：

```text
输入 GitHub PR 链接
  -> 获取 PR 元信息
  -> 获取变更文件和 patch
  -> 规则风险预检查
  -> AI Review 生成
  -> 页面展示
  -> Markdown 导出
```

后续工作重点应转向提升质量：减少误报、优化上下文、增强错误处理、支持私有仓库、增强报告可信度和可用性。
