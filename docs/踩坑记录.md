这个文件只记录开发中值得复盘的问题。普通命令备忘不放这里，除非它暴露了环境、流程或设计上的问题。

## 1. 本地存在多份仓库导致运行环境混乱

问题表现：

- 在一个目录里执行 `pip install -r requirements.txt`，提示找不到 `requirements.txt`。
- 在另一个目录里运行后端，提示 `No module named uvicorn`。
- 代码明明已经更新，但 IDE 打开的项目看起来还是旧状态。

原因：

- 本机同时存在 Desktop 和 code/PythonProject 下的两份项目拷贝。
- 实际开发和依赖安装一直发生在 Desktop 目录，IDE 一度打开了另一份旧拷贝。

处理方式：

- 统一以 `/Users/chenghe/Desktop/七牛云-AI PR Review 助手` 作为当前开发目录。
- 后端虚拟环境使用 `/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api/.venv`。
- 后续如果移动项目，先确认 Git remote、分支、依赖和 IDE 工作目录都指向同一份仓库。

后续约束：

- 遇到“文件不存在”或“依赖不存在”时，先检查 `pwd` 和 IDE 打开的项目路径。
- 不在多个拷贝之间手动粘贴代码，避免产生不可追踪的差异。

## 2. PyCharm 运行 FastAPI 时容易混淆模块、脚本和工作目录

问题表现：

- PyCharm 中找不到 `Module name: uvicorn` 选项。
- 直接运行时提示找不到 `uvicorn`。
- 对 `app.main:app --reload` 是什么产生疑惑。

原因：

- PyCharm 的运行配置界面不同版本展示不完全一致，有的入口叫模块，有的入口叫脚本。
- `app.main:app` 不是 Python 函数形参，而是 Uvicorn 的应用定位格式：`模块路径:应用变量名`。
- 后端必须以 `api` 目录作为工作目录，否则 Python 不一定能正确找到 `app` 包。

处理方式：

可靠命令是：

```bash
cd /Users/chenghe/Desktop/七牛云-AI PR Review 助手/api
./.venv/bin/python -m uvicorn app.main:app --reload
```

如果 PyCharm 没有模块运行选项，可以使用脚本路径：

- Python 解释器：`/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api/.venv/bin/python`
- Script path：`/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api/.venv/bin/uvicorn`
- Parameters：`app.main:app --reload`
- Working directory：`/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api`

后续约束：

- README 中的启动命令优先使用终端命令，IDE 配置作为补充说明。
- 后端运行问题优先检查解释器、工作目录和依赖安装位置。

## 3. Vite 默认端口被其他项目占用

问题表现：

- 当前项目希望使用 `http://localhost:5173`。
- Vite 发现 5173 被占用后，自动切到 5174。
- 前后端代理和浏览器访问地址容易因此不一致。

原因：

- 另一项目进程占用了 5173。
- Vite 会自动寻找下一个可用端口，但项目文档和代理配置默认仍按 5173 描述。

处理方式：

- 查找占用端口的进程并停止旧项目。
- 当前项目前端继续使用默认 5173，后端 CORS 也允许 5173。

后续约束：

- 手动测试前先确认浏览器打开的是当前项目的 Vite 地址。
- 如果临时使用 5174，需要确认 Vite proxy 仍然能把 `/api` 转发到后端。

## 4. PR 流程要区分“创建 PR”和“合并回 main”

问题表现：

- 开发完成后容易把“推送并创建 PR”和“本地合并回 main”理解成同一步。

原因：

- 创建 PR 是把功能分支提交给远端，方便检查和手动测试。
- 合并回 `main` 是在确认功能可用后，把分支内容正式进入主分支。

处理方式：

- 当前项目采用：先创建 PR，用户手动测试，通过后再合并回 `main`。
- 合并后重新跑测试和构建，再推送 `main`。

后续约束：

- 未经手动测试，不急着合并主分支。
- 每次合并后确认 GitHub PR 状态和本地 `main` 同步。

## 5. GitHub 公开 API 可以先跑通，但不能替代完整鉴权设计

问题表现：

- 公开 PR 元信息可以不配置 token 直接获取。
- 这容易让人误以为 GitHub 接入已经完整。

原因：

- GitHub 公开 API 适合早期 MVP，但存在限流。
- 私有仓库、企业仓库和更高频率请求都需要 token 或 GitHub App。

处理方式：

- 当前阶段只声明支持公开 GitHub PR。
- 后续支持私有仓库时，再引入 token 配置、权限说明和错误提示。

后续约束：

- 不把公开 API 的成功等同于生产级 GitHub 集成。
- 涉及 token 时必须通过环境变量配置，不能提交到仓库。
