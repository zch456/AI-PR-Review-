# 踩坑记录

这个文件只记录开发中真正值得复盘的问题。普通命令备忘不放这里，除非它暴露了环境、流程、架构或安全上的问题。

## 1. 本地多份仓库导致环境混乱

问题表现：

- 在一个目录执行安装命令，提示找不到 `requirements.txt`。
- 在另一个目录启动后端，提示找不到 `uvicorn`。
- IDE 打开的代码和实际开发目录不一致。

原因：

- 本机曾同时存在 Desktop 和 code/PythonProject 下的项目拷贝。
- 依赖安装、代码开发和 IDE 打开路径不一致。

处理方式：

- 统一以 `/Users/chenghe/Desktop/七牛云-AI PR Review 助手` 作为开发目录。
- 后端虚拟环境固定在 `api/.venv`。
- 遇到依赖或文件不存在时，先确认 `pwd`、Git remote 和 IDE 工作目录。

## 2. PyCharm 运行 FastAPI 的配置容易混淆

问题表现：

- PyCharm 找不到 `Module name: uvicorn`。
- 运行时提示 `No module named uvicorn`。
- 对 `app.main:app --reload` 的含义不清楚。

原因：

- PyCharm 不同版本的运行配置入口不完全一致。
- 后端必须使用正确的 Python 解释器和 `api` 工作目录。
- `app.main:app` 是 Uvicorn 的 ASGI 应用定位格式，不是函数形参。

可靠配置：

- Script path：`/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api/.venv/bin/uvicorn`
- Parameters：`app.main:app --reload`
- Working directory：`/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api`
- Python interpreter：`/Users/chenghe/Desktop/七牛云-AI PR Review 助手/api/.venv/bin/python`

## 3. Vite 端口占用导致访问错项目

问题表现：

- 期望使用 `5173`，但 Vite 自动切到 `5174`。
- 浏览器打开的页面不是当前项目。

原因：

- 其他项目占用了 Vite 默认端口。

处理方式：

- 手动测试前检查端口占用。
- 确认浏览器地址、Vite 输出和当前项目目录一致。

## 4. PR 状态和本地合并方式要统一

问题表现：

- 曾经本地 merge 后 push `main`，GitHub PR 显示 `CLOSED` 但没有 `mergedAt`。

原因：

- GitHub 没有通过 PR merge 操作完成合并，只识别到分支内容已进入主分支。

处理方式：

- 后续采用先创建 PR，再用 `gh pr merge` 合并。
- 合并后拉取 `main` 并重新跑验证。

## 5. API Key 不能出现在默认配置里

问题表现：

- AI 接入时，配置文件中曾出现硬编码 DeepSeek API Key 默认值。

风险：

- 一旦提交到远程仓库，就可能造成凭据泄露。

处理方式：

- `DEEPSEEK_API_KEY` 只从环境变量读取，默认空字符串。
- README 只写配置方式，不写真实 key。
- 提交前用 `rg` 扫描疑似密钥。

后续约束：

- 所有 token、key、secret 都必须走环境变量或本地未跟踪配置。
- PR 描述中也不要粘贴真实凭据。

## 6. 风险规则需要控制误报

问题表现：

- `package-lock.json` 中依赖名包含 `token` 时，曾被误判为疑似敏感信息。

原因：

- 初始规则只要新增行包含 `token`、`secret` 等关键词就报警，过于粗糙。

处理方式：

- 将敏感信息规则收紧为“新增行中出现敏感键的赋值”。
- 增加回归测试，确保依赖名中的关键词不触发敏感信息风险。

后续约束：

- 风险规则不是越多越好，要优先保证信号质量。
- 每次修误报都应补回归测试。

## 7. 模型名和文档必须保持一致

问题表现：

- README 和配置里一度使用了 `deepseek-chat`，但实际需要的是 `deepseekv4pro`。

处理方式：

- 修正 `DEEPSEEK_MODEL` 默认值。
- 同步 README 示例。
- 用 `rg` 检查旧模型名没有残留。

后续约束：

- 模型名、接口地址、环境变量示例要统一检查。
- 如果模型由用户指定，代码默认值和 README 都要清楚说明可覆盖。
