# 优化拓展

这个文档面向“项目更上一层楼”的下一阶段。当前项目已经完成初步闭环，后续重点应从“能跑通”转向“更准、更稳、更好用、更像真实开发工具”。

## 总体方向

下一阶段建议围绕四条主线推进：

- **分析更准**：减少误报和漏报，提升上下文质量。
- **体验更顺**：让用户更容易配置、等待、理解和导出结果。
- **工程更稳**：完善错误处理、测试、组件边界和安全边界。
- **场景更真实**：支持私有仓库、GitHub 评论、团队规则和多平台。

## 1. 上下文获取升级

当前模型主要依赖 PR 元信息、文件列表和 patch。下一阶段可以增强上下文：

- 获取 PR 描述正文和提交信息。
- 获取变更文件的完整内容。
- 获取变更行附近的函数或类上下文。
- 自动查找相关测试文件。
- 识别项目类型和构建配置，例如 `package.json`、`pyproject.toml`、`requirements.txt`。

建议实现方式：

- 增加 `context_collector` 模块。
- 将上下文拆成 `metadata`、`files`、`patches`、`relatedFiles`、`riskSignals`。
- 大 PR 按文件分组，先局部分析，再聚合总结。

预期收益：

- AI Review 不再只看 patch 片段。
- 对跨文件影响、测试缺失和配置风险判断更准确。

## 2. 模型调用链优化

当前是一次模型调用生成总结和 findings。后续可以做成多阶段流水线：

1. 快速摘要：先总结 PR 主题和主要变更。
2. 文件级分析：按高风险文件或文件组分析。
3. 聚合去重：合并相似发现，过滤低价值建议。
4. 最终报告：生成面向人的 Review 文本。

建议增加：

- 模型超时重试。
- JSON schema 校验。
- 低置信度问题进入 `reviewQuestions`，不直接进入 `findings`。
- 模型输出质量评分，例如是否有文件路径、是否有建议、是否过于泛泛。

预期收益：

- 大 PR 更稳定。
- 报告更像真实 Review，而不是一次性泛读。

## 3. 规则引擎可配置化

当前规则写死在 `risk_engine` 中。后续可以支持配置：

- 用户自定义高风险目录。
- 用户自定义忽略文件，例如 lock 文件或生成文件。
- 团队自定义关键词，例如支付、订单、权限、数据迁移。
- 不同项目类型的规则集，例如 Python、Node、前端、后端。

建议文件：

```text
review-rules.example.json
```

配置示例：

```json
{
  "highRiskPaths": ["api/app/auth", "payment", "migrations"],
  "ignoredPaths": ["web/package-lock.json"],
  "largeChangeThreshold": 300
}
```

预期收益：

- 更贴合真实项目。
- 用户可以降低误报，而不是只能接受默认规则。

## 4. GitHub 集成增强

当前只读取公开 PR。后续可以增强为真实工作流工具：

- 支持 `GITHUB_TOKEN`。
- 支持私有仓库。
- 支持读取 PR 评论和历史 Review。
- 支持把选中的 AI 建议发布为 GitHub review comment。
- 支持只评论新增行，避免评论到无效行号。

关键注意点：

- token 权限最小化。
- 后端不能把 token 返回给前端。
- 发布评论前必须让用户确认，避免 AI 自动刷屏。

预期收益：

- 从“本地辅助工具”升级为“可参与 GitHub Review 流程的工具”。

## 5. 前端体验升级

当前前端功能集中在一个 `App.tsx` 中，已经可用，但继续扩展会变重。

建议拆分组件：

- `PrInputPanel`
- `PrMetadataSummary`
- `RiskSignalList`
- `ChangedFileList`
- `AiReviewPanel`
- `MarkdownExportButton`
- `ErrorMessage`

体验优化：

- 分阶段加载状态：解析 URL、获取 GitHub 数据、执行风险规则、生成 AI Review。
- AI Review 生成失败后保留已有 PR 分析结果。
- 支持复制 Markdown，而不只是下载文件。
- 支持折叠长文件列表和风险列表。
- 支持“只分析高风险文件”开关。

预期收益：

- 页面更清晰。
- 代码更容易维护。
- Demo 更顺滑。

## 6. 测试和演示能力增强

当前后端测试已经覆盖主要模块。后续建议增加：

- 前端组件测试。
- Playwright 端到端测试。
- GitHub API fixture，避免演示依赖网络。
- AI provider fake server，用于稳定演示 AI Review。

建议增加演示模式：

```text
DEMO_MODE=true
```

演示模式下使用本地 fixture 返回固定 PR 数据和固定 AI Review，保证 Demo 视频稳定。

预期收益：

- 避免现场网络、限流、API key、模型响应波动影响展示。

## 7. 报告质量升级

当前报告包含总结和 findings。下一阶段可以扩展为：

- 关键变更摘要。
- 风险文件排序。
- 具体发现项。
- Review 问题。
- 推荐测试。
- 置信度说明。
- 分析范围说明。

推荐报告结构：

```json
{
  "summary": "...",
  "riskLevel": "medium",
  "keyChanges": [],
  "findings": [],
  "reviewQuestions": [],
  "recommendedTests": [],
  "analysisScope": {
    "includedFiles": 10,
    "omittedFiles": 2,
    "truncatedFiles": 1
  }
}
```

预期收益：

- 用户能判断 AI 看了什么、没看什么。
- Review 建议更可信。

## 8. 安全和隐私增强

后续如果支持私有仓库或团队使用，必须补齐安全边界：

- API key 和 GitHub token 加密存储，或只保存在本地环境变量。
- 不持久化 PR 代码内容，除非用户明确开启。
- 日志中隐藏 token、Authorization header 和敏感配置。
- README 明确说明哪些内容会发送给 AI provider。
- 对私有仓库显示醒目的隐私提示。

预期收益：

- 工具可以更放心地用于真实代码库。

## 推荐推进顺序

建议按以下顺序继续做：

1. `.env.example` 和配置状态提示。
2. 前端组件拆分。
3. Review 报告结构升级。
4. 上下文构建模块化。
5. 大 PR 分块分析。
6. GitHub token 支持。
7. GitHub 评论发布。
8. Demo mode 和 fixture。

这样推进的好处是：先稳住配置和维护性，再提升分析质量，最后接入更真实的 GitHub 工作流。
