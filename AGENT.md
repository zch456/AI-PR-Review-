# AI PR Review 助手协作约定

## 项目定位

本项目是一个网页端 AI PR Review 工具。用户输入 GitHub PR 链接后，系统获取 PR 信息和代码变更，并逐步生成变更总结、风险提示和 Review 建议。

当前代码已经完成：

- React + Vite 前端骨架。
- FastAPI 后端骨架。
- GitHub PR URL 解析。
- 公开 GitHub PR 元信息获取。
- 前端展示 PR 标题、作者、状态、分支、增删行和变更文件数量。

后续重点是把“能获取 PR 信息”推进到“能辅助 Review 代码”。

## 开发原则

- 保持小步迭代，每次只解决一个清晰问题。
- 新功能优先通过独立 PR 完成，不把无关重构、样式调整和功能逻辑混在一起。
- 主分支必须保持可安装、可运行、可验证。
- 涉及业务逻辑的修改按 TDD 推进：先写或调整测试，再实现，再跑验证。
- 文档跟随代码更新，避免文档描述和实际行为脱节。
- 依赖第三方库时，要在 README 或相关文档中说明用途。

## 分支与 PR 流程

- 从 `main` 创建功能分支，分支名使用 `codex/<主题>`。
- 每个 PR 聚焦一个功能或一组紧密相关的文档调整。
- PR 描述至少包含：做了什么、为什么这样做、如何验证。
- 合并前先跑自动化验证；涉及页面体验时再做一次手动检查。
- 用户手动测试通过后，再把功能分支合并回 `main` 并推送。

## 验证命令

后端测试：

```bash
cd api
./.venv/bin/python -m pytest -q
```

前端构建：

```bash
cd web
npm run build
```

如果改动只涉及文档，也要至少检查 Git 状态和文档路径，确认没有误改业务文件。

## 文档分工

- `README.md`：面向使用者，说明项目是什么、怎么运行、当前能力、依赖和后续计划。
- `docs/开发计划.md`：合并需求、设计和阶段计划，记录项目目标、架构思路、开发路线和验收方式。
- `docs/踩坑记录.md`：记录开发过程中真正值得沉淀的问题、原因、处理方式和后续约束。
